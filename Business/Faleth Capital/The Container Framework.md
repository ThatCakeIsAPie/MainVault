# The Container Framework

_A Modular, Scalable Organization Model_

Faleth Capital  
Internal Framework Document  
Version 1.0 — Draft

## Philosophy

Organizations grow complex, then brittle. Middle management bloats. Coordination fractures. Innovation stalls. The solution isn't more hierarchy—it's **modularity**.

Inspired by Docker containers: isolated units with defined inputs/outputs. Scale horizontally. Compose into systems. Simplify or split when complex. Each \"container\" (tribe/division) operates as its own profit center, trading services with others via gross-margin flows.

This framework makes Faleth subsidiaries anti-fragile: tribes self-organize under [[The Contribution Framework|Contribution Framework]], interfacing via clear contracts. No central chokepoints. Optimize locally, thrive globally.

## Core Principles

1. **Inputs/Outputs Defined**: Every tribe exposes simple APIs (services/products). No internal dependencies.
2. **Profit Centers**: Each runs its own COF/pool/points. Trade = revenue.
3. **Scale Modularly**: Add tribes for capacity. Merge/split on complexity.
4. **Self-Direction**: Elected Leads own tribe governance. Reps align at sub/office level.
5. **Flow Integration**: Contribution points track inter-tribe value (e.g., \"Parts Delivered\" process).

## Structure

```
Subsidiary
├── Tribe 1: Inputs → Outputs (e.g., Raw Materials → Components)
├── Tribe 2: Inputs → Outputs (e.g., Components → Assemblies)
├── Tribe 3: Inputs → Outputs (e.g., Assemblies → Finished Goods)
└── Orchestration Layer (minimal: contracts, standards, elected reps)
```

### Tribe Design (<150 people ideal; Semco-inspired cells)

- **Size**: 10-150. Split at coordination drag.
- **Autonomy**: Set points/COF/profit use (reinvest? Land buy?).
- **Interfaces**: Flowcharts for every input/output process.
- **Points Across Boundaries**: e.g., Tribe 1 earns \"Components Delivered\" points when Tribe 2 accepts.

### Manufacturing Example: Electronics Assembly

```
Tribe A: Sourcing (Inputs: Orders → Outputs: Parts Inventory)
  - Points: Vendor Negotiate, Inventory Stock, Quality Check

Tribe B: SMT Assembly (Inputs: Parts → Outputs: PCBs)
  - Points: Line Setup, Component Place, Reflow Solder, AOI Inspect

Tribe C: Final Assembly/Test (Inputs: PCBs → Outputs: Finished Units)
  - Points: Board Mount, Enclosure Fit, Functional Test, Pack/Ship

Inter-Tribe Trades:
- A → B: \"Parts Delivered\" (B credits points to A)
- B → C: \"PCBs Ready\" (C credits to B)
```

**COF Flywheel**: Fixed % of gross margin to pool. Simplify process (automate pick/place) → fewer total points → higher $ per point. Tribe wins.

## Governance

- **Intra-Tribe**: Full [[The Contribution Framework|Contribution]] autonomy.
- **Inter-Tribe**: Elected Lead reps vote on standards/contracts (e.g., \"PCB spec v2\"?).
- **Sub-Level**: Tribe reps + owner set sub-parameters.
- **Office**: Sub reps in republic model.

## Scaling Rules

| Trigger | Action |
|---------|--------|
| >150 people | Split tribe |
| Bottleneck at interface | Add buffer tribe (e.g., Quality Gate) |
| Redundant processes | Merge tribes |
| New product line | Spin greenfield tribe |

## Integration with Existing Frameworks

- **Contribution**: Points include inter-tribe processes.
- **Financial**: Tribe revenue → sub/office profit share.
- **Equity**: Tribe performance mints sub shares → office equity.

## Implementation Checklist

1. Map subsidiary processes to tribes (inputs/outputs/flowcharts).
2. Launch pilot tribe under Contribution Framework.
3. Define/test inter-tribe contracts (points/revenue).
4. Elect reps, run quarterly interface review.
5. Scale: Monitor complexity, split/merge dynamically.
